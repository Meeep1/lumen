# Lumen — Release Roadmap

Status snapshot as of 2026-07-02, updated 2026-07-04. This roadmap is grounded in the actual state of the codebase (not the spec's aspirations) — every item below reflects something confirmed by reading `app_spec.md`, the backend routes/schema, and the iOS views. Where the spec claims a feature exists and it doesn't, that's called out explicitly.

**Phase 1 progress: 9/9 code-complete, and every external blocker that was ever going to close is now closed.** 1.1-1.6 were already done; 1.8 is done too. 1.5's Apple Developer Portal capability is enabled and 2.1's real APNs key is configured and verified against Apple's real servers — both were the last non-code dependencies blocking those items. 1.7's real Terms/Privacy Policy blocker is resolved down to one `[INSERT]` fill-in (governing-law state — the one thing that genuinely needs a human, since it's wherever you personally live/operate) plus optional legal review. App Store Connect's nutrition-label submission itself is still a UI-only step outside this repo, not something code can close.

Priorities: **P0** = release blocker (cannot ship without it), **P1** = should ship in v1.0 but app is technically usable without it, **P2** = quality-of-life, do soon after launch, **P3** = future/differentiation.

## Phase 1 — Release Blockers (P0)

Nothing ships to TestFlight/App Store until this phase is done. Ordered roughly by dependency.

### 1.1 Real-time chat (SocketManager rebuild) — ✅ DONE (2026-07-03)
- Built differently than originally scoped: `socket.io-client-swift` couldn't be resolved via SPM in this dev environment (`xcodebuild -resolvePackageDependencies` never completed a resolution, likely needs an interactive Xcode session unavailable here). Swapped the whole real-time layer to a plain WebSocket instead — backend now uses `@fastify/websocket` (`backend/src/socket/handlers.ts` rewritten with a per-user socket registry replacing Socket.IO rooms), and `SocketManager.swift` uses Foundation's native `URLSessionWebSocketTask` — zero external dependencies.
- `connect()`/`disconnect()` wired to JWT-authenticated lifecycle (bearer token via request header), reconnect-with-backoff on drop, app-level keepalive ping every 30s (doubles as Redis online-status refresh).
- `sendMessage`/`typing`/`mark_read` and their listeners (`new_message`/`user_typing`/`messages_read`) all implemented and verified live with a two-client test — message delivery, typing indicator, and read receipts all round-trip correctly.
- Not yet done: explicit reconciliation of the REST send path (`match.ts`) vs. socket path beyond client-side dedup-by-id — works today, but worth a deliberate audit later.

### 1.2 Admin authorization — ✅ DONE (2026-07-03)
- `requireAdmin` middleware added (`backend/src/middleware/auth.ts`), checks `User.isAdmin` after JWT verification, applied to both `GET /reports/admin` and `POST /reports/admin/:reportId/action`.
- `npm run make-admin -- <email>` (and `--revoke`) added as the way to grant/revoke admin access.
- Verified: non-admin gets 403, admin gets 200.

### 1.3 Suspension enforcement — ✅ DONE (2026-07-04)
- `authenticate` middleware now fetches `isSuspended`/`suspendedUntil` on every request (not just login) and rejects with `403 {error, code: "ACCOUNT_SUSPENDED"}` if still active; auto-clears the suspension if `suspendedUntil` has already passed. Also fixed a related pre-existing bug found while testing this: the login route's own suspension check didn't account for expiry either, so anyone whose temporary suspension had already lapsed was permanently locked out (login was the only way to get a token, and nothing ever cleared `isSuspended` for them) — login now auto-clears expired suspensions the same way.
- `requireAdmin` now reuses the `isAdmin` flag `authenticate` already fetched (stashed on `request.userIsAdmin`) instead of a second DB query.
- iOS: `APIError.accountSuspended`, `ErrorResponse.code`, and a `.accountSuspended` `NotificationCenter` event so `APIService` (which has no reference to `AuthenticationManager`) can still trigger a forced logout + on-brand alert on `AuthenticationView` without a circular dependency.
- Verified via curl: active suspension blocks both login and mid-session requests with an already-issued token; expired suspension auto-clears on either path; admin gating still works post-refactor.
- **Not covered:** the two multipart upload methods (`uploadPhoto`, `submitVerificationPhoto`) don't go through the shared `request()` method, so they won't trigger the client-side forced-logout flow specifically (they'll still get the 403 from the server, just surfaced as a generic error rather than the dedicated suspension alert).

### 1.4 Photo moderation — ✅ DONE (2026-07-04), revised same day
- First pass wired up AWS Rekognition gated behind `NODE_ENV=production` — but that meant local dev kept auto-approving *everything*, including actually-NSFW test uploads, which is how this got caught (user uploaded NSFW content locally and it sailed through). Replaced Rekognition entirely with **NSFWJS** (MobileNetV2 trained for NSFW classification) running on plain `@tensorflow/tfjs` + `sharp` for image decoding — no AWS, no API keys, no per-image cost, classification runs on the server's own CPU, and it's active in dev and prod alike now (no environment gate at all). `@tensorflow/tfjs-node` (the native-bindings version) failed to compile in this environment (missing Xcode headers), so this uses the pure-JS tfjs backend instead — slower per-classification (~150-250ms) but zero native build step.
- Same three-tier policy as before, now driven by NSFWJS's Porn/Hentai/Sexy probability scores instead of Rekognition labels: high-confidence explicit → `rejected` outright, borderline explicit or high-confidence suggestive → `pending` (queued for a human via the admin site's Photos tab), otherwise → `approved`. Thresholds tunable via `MODERATION_REJECT_SCORE`/`MODERATION_PENDING_SCORE`/`MODERATION_SEXY_PENDING_SCORE` env vars (defaults 0.85/0.4/0.7).
- Model loads once at server boot (~1-2s, added to the existing Prisma/Redis warm-up in `server.ts`) rather than on the first upload. One-time network fetch of the model weights (~5MB, from NSFWJS's public GitHub-hosted default) on first load each process start — no per-request network calls after that.
- `backend/src/routes/moderation.ts` (`GET /moderation/photos?status=`, `POST /moderation/photos/:photoId/action`) and admin site's "Photos" tab unchanged from the first pass — they just read/write `Photo.moderationStatus`, agnostic to what set it.
- Verified: real classification against a real photo (correct probabilities returned), threshold branching logic (6 cases covering all three outcomes), local-dev upload actually runs moderation now (no more blanket auto-approve), and the admin approve/reject queue still works end-to-end.
- **Update (2026-07-04, later same day):** the "not verified against real NSFW" gap above got closed for real — user reported an actual NSFW upload sailed through, which is exactly the scenario that couldn't be tested synthetically. Root-caused for real this time:
  - NSFWJS ships two bundled models. The default (`MobileNetV2`) scored the real flagged photo at only ~5% combined Porn+Hentai — nowhere near the 0.4 pending threshold. Switched to the bundled `InceptionV3` model, which scored the *identical* photo at ~17% — a 3x stronger signal on the same input, confirmed via direct side-by-side testing (not just theory). Verified no NaN/preprocessing bugs first (pixel data, tensor construction, and normalization all checked correct) before concluding it was a model-choice problem, not a pipeline bug.
  - Recalibrated thresholds down to match the score ranges these models actually produce (clean photos: ~3-5% combined explicit; the one confirmed-bad photo: ~17%): reject 0.85→0.5, pending 0.4→0.12, sexy-pending 0.7→0.15. Explicitly flagged in the code as a first-pass recalibration against a sample size of one, not a settled result.
  - Re-uploaded the exact same real photo through the actual endpoint post-fix: now correctly lands on `pending` instead of `approved`.
  - Added `POST /moderation/rescan` (admin site "Rescan Approved Photos" button) so photos approved *before* this fix — the "it's still there" half of the complaint — get re-checked, not just new uploads. First version of this **froze the entire server for other users** (single-threaded Node, each classification is 5-12s of blocking synchronous CPU work, and it was churning through 200+ mostly-synthetic seed-test photos); had to kill the stuck process mid-run. Fixed by excluding seed/test accounts from rescan (they're synthetic images we control, not real content) and adding an event-loop yield between each photo. Re-run against the real account: found and flagged all 4 previously-"approved" photos (1 auto-rejected at 98% Hentai, 3 sent to pending review) — confirms the original bug was real and this catches it.
  - Also found and fixed while investigating: the admin site's "Failed to load photo queue" was a red herring — `@fastify/rate-limit`'s 100 req/min (shared per source IP) was getting exhausted by normal admin-site + testing traffic, not a real endpoint bug. Bumped to 500/min for local dev.
  - **Resolved 2026-07-04: moved to a background job queue (see 2.7).** This used to say every photo upload cost 5-12s of *blocking* server time, freezing the app for every other user during that window. It now returns in milliseconds and a separate worker process handles the actual classification — see 2.7 for the full change.
  - **Update (2026-07-04, later still):** two more rounds of real-world threshold tuning happened after the above. (1) `GET /profile/me` was found to filter to `moderationStatus: 'approved'` same as viewing someone else's profile — so any of *your own* pending photos just vanished with no explanation ("seems to not be added"). Fixed: your own profile now shows every photo regardless of status, with the status included so the app can badge it. (2) The pending threshold (0.12) turned out to flag nearly every real upload, so it got raised to 0.35 — which then overcorrected the other way: a real NSFW photo (rear nudity, scored ~25% combined Porn+Hentai) sailed through as approved because 0.35 was well above it, while two confirmed-clean real photos scored only ~0.1-0.2%. The clean/bad gap was much wider than assumed, so pending dropped to **0.15** — verified against all 4 confirmed real examples now on hand (2 clean, 2 bad) and it correctly separates all of them. Also added photo management UI (`ManagePhotosView.swift`, `PhotoCropView.swift`) — reorder, delete, and a crop step before upload — closing out Phase 2.3 early, since it was directly requested alongside these bug reports.
  - **Update (2026-07-04, drawings + appeals):** user reported real art/drawings were being auto-rejected outright, and asked for those to be allowed plus a way to appeal a fully-removed photo. Added a `drawingLeewayThreshold` (`MODERATION_DRAWING_LEEWAY_SCORE`, default 0.15) that exempts a photo from the explicit-reject path if NSFWJS's own `Drawing` score clears it — the first version of this check (`drawingScore > explicitScore`) was mathematically dead code, since all five NSFWJS class scores sum to ~1 and drawingScore is capped at ≤0.25 exactly when the reject threshold (≥0.75) can fire; caught by working through the math before calling it done, fixed with an absolute floor instead. Manual admin rejection and the rescan reject path no longer delete the file/row either — they set `moderationStatus: 'rejected'` and (manual only) `rejectedByAdminId`, which is what distinguishes a human's decision from the automated one and gates appeal eligibility per the user's explicit instruction ("only allow appeals for auto-removed ones, not ones already looked at by a person"). New `POST /profile/photos/:photoId/appeal` (blocked if `rejectedByAdminId` is set or an appeal is already pending) and admin `GET /appeals` / `POST /appeals/:photoId/action` (approve restores `approved`; deny sets `rejectedByAdminId`, closing the door on re-appealing). iOS: `ManagePhotosView` shows "Appeal this decision" / "awaiting review" / "Reviewed by our team" depending on status, with an `AppealComposerView` sheet for an optional message.

### 1.5 Sign in with Apple — ✅ DONE (2026-07-04)
- Scoped deliberately: Apple only replaces the *password* step, not the rest of this app's signup requirements (phone/OTP, age, gender identity, fem attestation all still apply either way) — the alternative (letting Apple's identity alone create a fully-formed account) would have meant making `dateOfBirth`/`genderIdentity`/`phone` nullable throughout the schema, which ripples into nearly every route and the iOS `User` model. Not worth that blast radius for what this app actually needs Apple Sign In for.
- Schema: `passwordHash` and (new) `appleUserId` are both nullable/unique. `POST /auth/apple` (login/link path) and `POST /auth/signup` (new field `appleIdentityToken`) both **re-verify the Apple identity token server-side** via `apple-signin-auth` — deliberately not trusting a client-supplied Apple user ID directly, since that would let anyone claim any identity string with no proof of ownership.
- Flow: tap "Sign in with Apple" → if an account already exists for that Apple ID (or a matching email, auto-linked) → straight to login. If not → `AppleSignInOutcome.needsSignup` routes to the same signup form, pre-filled with whatever email Apple provided, password fields hidden.
- iOS: `SignInWithAppleButton` used as-is, no custom styling — Apple's Human Interface Guidelines require their standard button for this specific control, unlike the custom-UI work done elsewhere in the app.
- Verified: invalid/malformed identity tokens correctly rejected (401) on both the login and signup paths; existing email/password login confirmed unaffected (regression check).
- **Update (2026-07-04): the Apple Developer Portal capability is enabled.** "Sign In with Apple" is now turned on for the bundle ID (done directly by the account holder in the portal — not something committable to this repo). The one remaining piece is real on-device testing with an actual Apple ID, which needs a physical device rather than a simulator.

### 1.7 App Store compliance pass — code-complete (2026-07-04), one small external fill-in remains
- **Age gate — found and fixed a real bug while verifying this.** The DOB check in `signupSchema` (validation.ts) mixed a UTC-parsed birth date with local-timezone getters (`.getFullYear()`/`.getMonth()`/`.getDate()`). Confirmed concretely: `new Date('2008-07-05T00:00:00Z').getDate()` returns `4`, not `5`, in US Eastern — any server timezone behind UTC reads a midnight-UTC DOB as the previous evening. Verified end-to-end against the live server: someone turning 18 *tomorrow* was being accepted *today*. Fixed by switching to `getUTC*()` consistently on both sides of the comparison in all three places this logic existed (`validation.ts`'s signup gate, plus the cosmetic `calculateAge()` duplicates in `swipe.ts`/`profile.ts`). Re-verified: exactly-18-today accepted, one-day-shy rejected, day-after accepted.
- **Terms of Service / Privacy Policy / Community Guidelines — done (2026-07-04), real hosted pages now exist.** Previously flagged as a hard external blocker ("has to come from you or counsel, not be fabricated") — resolved pragmatically: drafted real, substantive documents grounded directly in the actual data-collection audit below (not generic boilerplate), and built a small public marketing site to host them at `backend/public/site/` (`index.html`, `privacy.html`, `terms.html`, `community-guidelines.html`, shared `style.css` matching the app's actual color palette), served by the same backend at `/`, `/privacy`, `/terms`, `/community-guidelines`. `SettingsView.swift`'s three `Link(destination:)` URLs now point at these real pages instead of `example.com`. **Two things still need a human, not code:** both documents have `[INSERT ...]` placeholders for the legal entity name and governing jurisdiction — the one part of "real ToS/Privacy Policy" that genuinely can't be fabricated — and a lawyer should review both before real launch, especially the liability/dispute-resolution sections. Everything else (what's collected, retention, moderation policy, safety language) is accurate to how the app actually works today, not aspirational copy.
- **Privacy nutrition label** — audited what's actually collected, for whoever fills out App Store Connect's questionnaire (a UI there, not something committable to this repo): email, phone number, date of birth, gender identity, precise location (lat/long, used to compute distance — only city/distance ever shown to other users), photos, verification selfie (reviewed by a human, not retained beyond review per current flow — confirm retention policy matches before submission), bio/prompt text, push notification token. No third-party analytics/ad SDKs exist in the codebase today. This exact list is now also what the new Privacy Policy page itself states, so the two stay consistent.
- **Account deletion discoverability** — confirmed fine as-is: `SettingsView.swift` has an always-visible red "Delete Account" button, not buried in a submenu. No change needed.
- **Content moderation + blocking/reporting** — already covered by 1.2/1.3/1.4/1.6; no new gaps found.
- **App icon — found and fixed a real submission blocker (2026-07-04).** App Store Connect rejected the build outright: no 120×120 (iPhone) or 152×152 (iPad) icon, and no `CFBundleIconName` key. Root cause was total — this project had no `Assets.xcassets` catalog at all, in any form, since it was first set up. Added `Lumen/Assets.xcassets/AppIcon.appiconset` using Xcode's single-size universal format (one 1024×1024 PNG, no alpha channel; Xcode's asset compiler generates every required size at build time), wired `ASSETCATALOG_COMPILER_APPICON_NAME` into both build configs via the `xcodeproj` gem (this project's classic, non-synchronized `.pbxproj` group structure makes hand-editing risky). Verified in a real device-target build: 120×120/152×152 files present at those exact pixel dimensions, `CFBundleIconName` populated correctly. **Note: placeholder artwork, not final branding** — a gradient-and-heart mark generated programmatically to unblock submission; swap `AppIcon1024.png` for real artwork before shipping (no project changes needed to do so).
- **Export compliance** — confirmed the iOS app uses no encryption beyond standard HTTPS (`URLSession`) and Keychain, both Apple-provided OS frameworks — no `CryptoKit`/`CommonCrypto`/custom crypto anywhere in the codebase. Qualifies for App Store Connect's standard exemption ("None of the algorithms mentioned above" on the encryption-documentation question); worth adding `ITSAppUsesNonExemptEncryption = false` to Info.plist so future submissions skip the manual question entirely (not yet done).
- **Remaining before submission:** fill in the two `[INSERT ...]` placeholders (legal entity name, governing jurisdiction) in the new ToS/Privacy pages, ideally with legal review; swap the dev-LAN-IP URLs in `SettingsView.swift` for the real production domain once Phase 3's domain/TLS work lands; fill out App Store Connect's nutrition label using the audit above.

### 1.9 Public site + hardened admin panel — ✅ DONE (2026-07-04)
- Built `backend/public/site/` (landing page + the three legal pages above) and mounted it at the backend's root (`/`) via a third `@fastify/static` registration in `server.ts`, alongside the existing `/uploads/` and `/admin/` ones — same backend, same deploy, but a completely separate, unauthenticated, public surface with zero links to `/admin/` anywhere in it.
- **The admin panel itself is now hidden behind an outer HTTP Basic Auth gate**, on top of (not instead of) the existing per-account `isAdmin` JWT check every real admin action already required. Before this, `/admin/`'s static HTML shell — the login page itself, plus whatever the page source reveals about the tech stack — was reachable by anyone who found or guessed the URL, even though they could never authenticate past it. New `middleware/basicAuth.ts`'s `requireBasicAuth` fails closed (500, not silently open) if `ADMIN_BASIC_AUTH_USER`/`PASSWORD` aren't set. Applied at two different granularities depending on the route file's shape: wrapped around the whole registration for `admin-tools.ts`/`moderation.ts` (admin-only in their entirety), but added directly into just the specific `/admin`-prefixed routes within `report.ts`/`verification.ts` (which mix real public user-facing routes with a couple of admin sub-paths — blanket-gating the whole file would have locked real users out of filing reports/submitting verification selfies).
- Verified live against the real running server: `/admin/` with no credentials → `401` with a proper `WWW-Authenticate` header (a real browser will show its native basic-auth prompt); with the wrong password → `401`; with the right credentials → `200` (then still requires the real admin login to do anything); `/admin-tools/*` → `401` unauthenticated. Public site paths (`/`, `/privacy`, `/terms`, `/community-guidelines`) → `200`, no auth needed, confirming the two surfaces are properly independent.
- **Found and fixed a real gotcha while wiring this up:** `.env` file edits don't trigger `tsx watch`'s auto-restart (only source file changes do) — the running dev server kept using the *old* JWT secrets and had no `ADMIN_BASIC_AUTH_*` values in memory at all for several verification attempts (every check came back `500`, "not configured") until the process was manually restarted. Worth remembering for any future `.env`-only change: `tsx watch` won't pick it up on its own.
- **Also fixed in the same pass:** `JWT_ACCESS_SECRET`/`JWT_REFRESH_SECRET` in `.env` were still the literal placeholder `GENERATE_SECRET_HERE` (flagged as a finding in the previous roadmap update, not yet fixed until now) — replaced with real `openssl rand -base64 32` values. Rotating these invalidates every previously-issued token, which is expected and harmless in dev (confirmed the review account can still log in fine and get a fresh token); do this deliberately in production, since it forces every logged-in session to re-authenticate.

### 1.8 Crash reporting & basic observability — done (2026-07-04)
- **Backend:** real Sentry (`@sentry/node`, see `src/sentry.ts`) — `Sentry.setupFastifyErrorHandler` reports every 500-level route error, plus new `process.on('uncaughtException'/'unhandledRejection')` handlers (there were none before — a bug outside any route's try/catch would previously crash silently with zero visibility). No-ops cleanly if `SENTRY_DSN` is unset. Added `ecosystem.config.js` for `pm2 start` in production (autorestart, log files) — dev still uses `npm run dev`/tsx watch, this is deploy-only.
- **iOS — dependency-free, deliberately.** Tried adding the Sentry Cocoa SPM package the same way 1.1 tried `socket.io-client-swift`: `xcodebuild -resolvePackageDependencies` hangs indefinitely regardless of which remote package is added (confirmed again, not package-specific — this environment's Xcode package resolution just doesn't complete, full stop). Reverted the package reference and built a small dependency-free reporter instead (`CrashReporter.swift`): `NSSetUncaughtExceptionHandler` for uncaught exceptions (full context — name/reason/stack) and POSIX signal handlers (SIGABRT/SIGILL/SIGSEGV/SIGFPE/SIGBUS/SIGTRAP) for fatal crashes that never raise an NSException, writing to a pending-crash file that gets shipped to the backend on next launch. Also wired up `MXMetricManagerSubscriber` (`MetricKitReporter.swift`) as a zero-dependency supplementary signal (OS-batched, arrives with a delay, but genuinely free). Both funnel into the same Sentry project via a new unauthenticated `POST /diagnostics/report` (rate-limited tighter than the app default — see `routes/diagnostics.ts`), since a crash can happen while logged out.

### 2.4 Automated testing — backend done (2026-07-04), iOS still open

### 2.5 Rate limiting & abuse prevention — ✅ DONE (2026-07-04)
- Confirmed a global default already existed (`@fastify/rate-limit`, `server.ts`, Redis-backed, 100 req/min per source by default) but nothing endpoint-specific — 100/min is far too loose for the four cases this item actually names (e.g. 100 login attempts/min per IP is a real brute-force budget). Added per-route overrides via Fastify's `config: { rateLimit: {...} }` route option, tighter than the global default: `POST /auth/signup` 5/min, `POST /auth/verify-otp` 10/min, `POST /auth/resend-otp` 3/min (the tightest — it's a real email send with no other gate in front of it), `POST /auth/login` 10/min, `POST /reports` 15/min, `POST /swipe` 60/min (generous — roughly one swipe/second sustained — since real rapid swiping is legitimate usage; the ceiling is for bot/scraping traffic, not real users).
- Verified live, not just by reading the code: fired 12 rapid `POST /auth/login` requests at the running dev server — the first 10 came back `401` (wrong credentials, as expected), the 11th and 12th came back `429`, confirming the limit actually engages at the configured threshold rather than just being present in the config.

### 2.6 Signed/expiring photo URLs — audited (2026-07-04), one real gap found and fixed
- **The enumeration concern itself doesn't hold up.** Photo paths are `/uploads/{photos|verification}/{userId}/{crypto.randomUUID()}.jpg` — both the directory (a Prisma `@default(uuid())`) and filename are cryptographically random v4 UUIDs (~122 bits of entropy each), not sequential IDs. That's not "trivially enumerable" in any practical sense; the real gap turned out to be somewhere else.
- **Found instead: account deletion never removed the actual files.** `DELETE /account` only ever deleted the `User` row — Prisma's `onDelete: Cascade` cleans up every *related row* (photos, swipes, matches, etc.) but was never going to touch a file sitting on disk, since nothing in that cascade is aware of the filesystem. Combined with the static `/uploads/` route having zero ownership/auth check (by design, for the local-dev fallback), this meant a "deleted" account's photos — including the verification selfie, arguably the most sensitive image type in the app — stayed servable forever at their old URL to anyone who'd saved it, completely defeating the point of a privacy-motivated deletion feature.
- Fixed with `deleteAllUserPhotos()` (`utils/storage.ts`), called from the account-deletion route right after the Prisma delete: recursively removes both the user's `photos/{userId}/` and `verification/{userId}/` directories. Verified end-to-end against the real running dev server (not just unit-level): signed up a throwaway account, uploaded a real photo, confirmed the file on disk, deleted the account via the actual API, confirmed the whole directory was gone.
- **Still open, deferred to the S3 migration (Phase 3) as originally scoped:** these are still plain unauthenticated static URLs, not short-expiry signed ones — anyone who has a *still-live* photo's URL (e.g. a match who saved it before either account was deleted) can fetch it with no auth check. Given the enumeration risk was the specific worry named here and that's now confirmed not to be real, and given real signed URLs are a S3-feature (`getPresignedUrl()` in `storage.ts` already has the seam for this — it's a one-line swap once bucket credentials exist), doing more than this locally would mean building expiring-token infrastructure for a local-disk mode that's getting replaced anyway.

### 2.7 Photo moderation background queue — ✅ DONE (2026-07-04)
- **The actual scaling ceiling for real users wasn't server size, it was one blocking call.** Raised while discussing cheap hosting options: NSFWJS classification (1.4) is 5-12 seconds of genuinely *blocking* synchronous CPU work with no native/GPU acceleration in this environment, and it ran inline on the upload request. Since Node is single-threaded, that meant every other user's swipes, messages, and page loads froze for that entire window whenever *anyone* uploaded a photo — a bigger box doesn't fix a blocking-call architecture problem, it just makes the freeze slightly shorter.
- **Fixed with BullMQ**, a Redis-backed job queue (Redis was already in the stack for rate-limiting/online-status). New `src/queue.ts` (queue definition) and `src/worker.ts` (a standalone entrypoint, run as its own `pm2` process — see `ecosystem.config.js`, now two apps instead of one). `POST /profile/photos` now: saves the file, creates the `Photo` row (defaults to `pending`, same schema default as before), enqueues a job, and returns immediately — no more waiting on classification at all. The admin site's "Rescan Approved Photos" button was the other blocking-loop offender (previously mitigated with an event-loop yield between photos, which only helped *between* iterations, never during one) — it now just enqueues a job per photo instead of processing inline, so a rescan of hundreds of photos can never stall the API either.
- **Cross-process notification required Redis pub/sub, not a direct call.** The worker runs in a genuinely separate OS process specifically so it can't block the API's event loop — which also means it has no access to the API process's in-memory WebSocket registry (`userSockets` in `socket/handlers.ts`) to tell a user their photo was reviewed. Bridged with Redis pub/sub: the worker publishes to a `photo-reviewed` channel, and the API process (which does hold the live sockets) subscribes and relays it via the exact same `sendToUser` path the admin's manual approve/reject already used — so the client sees identical behavior regardless of which side made the decision.
- **Notification logic is transition-based, not outcome-based** — this one job type now backs both a brand-new upload (always starts from `pending`) and an admin rescan of an already-approved photo, so "just reached `approved`" isn't enough to decide whether to notify (a rescan confirming an already-approved photo stays approved would otherwise fire a pointless "your photo was approved" notification for a photo that's been live for weeks). Only fires when the status actually *changes* to a final `approved`/`rejected` verdict; landing on `pending` never notifies, matching the reasoning the original rescan code already had for that case.
- **Verified for real, not just "compiles and doesn't crash":** timed an actual upload through the real running dev server — the request returned in **23ms** (previously 5-12 *seconds*), landed on `pending`, and the DB showed a correct `approved` verdict with real classifier labels ~8 seconds later once the separate worker processed it. Then proved the actual point of this change: fired a photo upload and, while it was still mid-classification, hammered `/health` five times in a row from a separate process — all five came back in ~8ms each, completely unaffected. Under the old architecture those would have been stuck behind the upload for the full classification window.
- **Effort note for whoever reads this later:** running the worker is not optional now — `npm run dev:worker` locally (alongside `npm run dev`), or the `lumen-worker` pm2 process in production (`ecosystem.config.js` starts it automatically alongside `lumen-backend`). If the worker isn't running, uploaded photos will sit on `pending` forever, since nothing else drains the queue.
